Forming Dept. Daily Production Report

Role: Developer  ·  Scope: 2 Department Sides, 62 Shift Sheets  ·  Tools: Excel · Power Query · MES Web API

The Problem

Supervisors had to physically walk to every machine, manually record production numbers, and enter that data into an outdated workbook with no automation — twice a day, every day. No connection to live systems, no department-level visibility.

How It Was Built

I built an analysis table inside the MES that exports as CSV. Power Query connects via the MES web API, pulling production data from all machines simultaneously. The workbook contains 62 sheets — one per shift, two per day — covering a full month. All API calls are built into a single parameterized M function so each sheet passes its own date and shift parameters into the same underlying logic. Metadata tables layer in department and machine context the raw API output couldn't provide.

The Outcome

Supervisors no longer walk the floor to collect numbers. Production data for every line in both sides of the forming department pulls automatically on refresh. The report also serves as a shift turnover tool — supervisors log notes, flag issues, and hand off context shift to shift.

Technical Highlights
  • Web API connection via Power Query pulling live MES data across all machines simultaneously
  • Single parameterized M function drives all 62 sheet queries — one place to maintain
  • Metadata tables enrich raw API output with department structure and machine context
  • OEE and production totals calculated at line level and aggregated by department side

Press Production Report & Speed Loss Initiative

Role: Developer  ·  Scope: Full Press Department  ·  Tools: Excel · Power Query · ERP · Operator Production Books

The Problem

No reliable way to track material consumption or approximate scrap at the run level. Speed loss was going completely uncaptured, making it impossible to identify whether slowdowns were tied to a specific job, deck, ink color, operator, or shift pattern.

How It Was Built

The report connects to ERP via Power Query to pull run-level data for every press. Combining linear feet produced with roll count and starting linear footage yields a scrap approximation per run. For speed loss, I overhauled the department's operator production books to add structured capture fields, then joined that data to the ERP run data for trend analysis.

The Outcome

Run-level visibility into linear feet produced, rolls consumed, and approximate scrap — all pulled automatically from ERP. The overhauled production books enabled trend analysis across job, deck, color, machine, operator, and shift.

Technical Highlights
  • ERP integration via Power Query pulling run-level linear feet, roll count, and starting footage per press
  • Scrap approximation calculated at the run level by combining ERP output with roll starting footage
  • Production book overhaul added structured speed loss fields with minimal operator burden
  • Speed loss joined to ERP run data — multi-dimensional trend analysis

Centerline Effectiveness Measurement System

Role: Developer  ·  Scope: Facility-Wide Machine Fleet  ·  Tools: Excel · Power Query · MES · Historical OEE Data

The Problem

Centerline events — resetting a machine to a known baseline mechanical state — are a significant investment of time and labor. Without a structured way to measure before and after OEE, there was no reliable way to know whether a centerline actually moved the needle.

How It Was Built

An anchor date is recorded when a centerline is completed — the dividing line between before and after periods. Large sets of historical OEE data are pulled from the MES for every machine in the fleet. Power Query automates all transformation using heavy grouping, indexing, and cross-dataset compilation to align pre- and post-centerline windows per machine. Everything runs on refresh with no manual data manipulation.

The Outcome

For the first time, the facility has a repeatable, data-driven way to evaluate the effectiveness of centerline events — specific OEE deltas tied to specific interventions, building a historical record that informs future improvement efforts.

Technical Highlights
  • Anchor date system records centerline completion dates as the before/after dividing line
  • Grouping, indexing, and cross-dataset compilation align pre/post windows per machine
  • All transformations automated via Power Query — no manual manipulation on refresh
  • Output normalizes comparison windows so OEE deltas are consistently structured

Machine Sensor Fault Analysis Report

Role: Developer  ·  Status: In Progress  ·  Tools: Excel · Power Query · MES Sensor Data · Web API

The Problem

High fault counts drive downtime and can signal conditions leading to defects. But very few or zero faults may indicate a sensor reliability issue. Both extremes were going undetected.

How It Was Built

Sensor fault data is pulled from the MES via Power Query and the web API. The data is transformed to count fault occurrences per machine broken down by fault type, organized into time-based sequences. The report flags machines at both ends — high counts for mechanical review, low/zero counts as potential sensor reliability concerns. Threshold values are currently being validated against known machine histories.

The Outcome

Maintenance and CI will have a tool to proactively identify two categories of risk that were previously invisible — machines stopping excessively, and machines where sensors may have stopped doing their job.

Technical Highlights
  • MES sensor fault data pulled via Power Query and web API
  • Two-tailed flagging: high counts flagged for review, zero counts flagged for sensor reliability
  • Fault frequency tracked over time to surface behavioral trends per machine
  • Side-by-side machine comparison for fleet-wide prioritization

Cross-System Reporting & Data Integration

Role: Developer  ·  Tools: Excel · Power Query · BI Tools · Access · MES · QMS · ERP

The Problem

MES, QMS, and ERP each held a piece of the picture but no one was looking at all three together. Cross-system insights required manual exports and hours of reconciliation.

How It Was Built

Using ODBC connections and API responses, Power Query pulls data directly from all three source systems into a single Excel layer. Transformation logic joins, cleans, and shapes the data into consistent output tables that feed dashboards. Reports refresh against live data — no manual exports, no copy-paste reconciliation.

The Outcome

Leadership has a fuller picture of performance, quality, and operational efficiency in one place. Insights that required hours of manual work are available on demand.

Technical Highlights
  • ODBC connections and REST API responses pulling live data from MES, QMS, and ERP simultaneously
  • Access databases used as intermediate query layer for legacy data sources
  • Designed for maintainability — no developer required to run or refresh reports

Plant Spoilage & Production Correlation Report

Role: Developer  ·  Tools: Power Query · SQL · MES API · ERP

The Problem

The plant was tracking spoilage at a surface level — units were being scrapped but there was no reliable way to know where in the plant it was coming from or what was driving it. The ERP and MES weren't connected and held complementary pieces of the answer.

How It Was Built

Connected the ERP via SQL and the MES via its REST API, then handled all ETL in Power Query using custom scaffolding and index key merges to join the two datasets. Built a 30-day trending view that breaks spoilage down from plant level to individual departments and correlates it against production volume.

The Outcome

Leadership went from almost no visibility into spoilage patterns to a real-time report they use to identify problem areas, investigate root causes, and make data-driven decisions about where to focus improvement efforts.

Technical Highlights
  • SQL connection to ERP and REST API connection to MES in a single Power Query flow
  • Custom scaffolding and index key merges join datasets that share no native key
  • 30-day trending view with plant-to-department drill-down
  • Production correlation layer identifies whether high spoilage tracks with higher output
